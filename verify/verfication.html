Verification Framework Description

R-T6: The car shall only open doors when there is a pending call:
  • Whenever the car is in MOVING state (judged by whether there is any atFloor sensor is true):
    If any atFloor sensor is true, the system shall calculate the CurrentFloor variable(floor in the code) and compare with the pendingCall variable to see whether there is a pending call at the floor and hallway stopped. The elevator will enter the VALID_STOP state when there is a matching pending call for the floor and hallway stopped, and INVALID_STOP otherwise.
  • Whenever the car is in VALID_STOP state:
    If all atFloor sensor are false and hence the internal CurrentFloor variable (floor in the code) is calculated to be 0, the car shall enter the MOVING state.
  • Whenever the car is in INVALID_STOP state:
    If all atFloor sensor are false and hence the internal CurrentFloor variable (floor in the code) is calculated to be 0, the car shall enter the MOVING state.


R-T7: The Car shall only open Doors at Hallways for which there are pending calls.
      To enforce this requirement, this monitor constructs 3 state machines.
  • CallState: This class keeps track of if any hall or car calls are pending at current floor.
  • FloorState: This class keeps track of the current floor (most recent atFloor message).
  • DoorState: This class keeps track of the state of the car doors: open or closed.
  • When the elevator doors open, this monitor checks if any of the hall or car button lights for that floor are on. If so, then there are pending calls at the floor. If not, then this monitor throws a warning.


R-T8.1: If any door is open at a hallway and there are any pending calls at any other floor(s), a Car Lantern shall turn on.
R-T8.2: If one of the car lanterns is lit, the direction indicated shall not change while the doors are open.
  • If door is open with pending calls and:
    • there is no lantern lit or both lanterns lit, the state shall move to INVALID_LOADING
    • there is a single lantern lit, the state shall move to VALID_LOADING
  • If door is open without pending calls and:
    • any lanterns are lit, the state shall move to INVALID_LOADING
    • no lanterns are lit, the state shall move to VALID_LOADING
  • Whenever the state is INVALID_LOADING or VALID_LOADING, if the lanterns change, the state shall move to LANTERN_CHANGED.


R-T8.2 (helper state machine):
  • This state machine stores the information whether the current stop is triggered by a hall call or car call.


R-T8.3: If one of the car lanterns is lit, the car shall service any calls in that direction first.
  • If there are no lanterns or both lanterns lit up, the shall both belong to state DO_NOT_CARE_STATE
  • If only UP lantern is lit up:
    • if desired floor is above current floor or is equal to INVALID_FLOOR, it shall result in VALID_DESIRED_FLOOR state.
    • if desired floor is below current floor, it shall result in INVALID_DESIRED_FLOOR.
  • If only DOWN lantern is lit up:
    • if desired floor is below current floor or is equal to INVALID_FLOOR, it shall result in VALID_DESIRED_FLOOR state.
    • if desired floor is above current floor, it shall result in INVALID_DESIRED_FLOOR.
  • Any lantern changes from VALID_DESIRED_FLOOR or INVALID_DESIRED_FLOOR shall result in the initial DO_NOT_CARE_STATE state.


R-T9: The Drive shall be commanded to fast speed to the maximum degree practicable.

      The monitor enforces this by constructing a state machine for the Drive.
  • Normal states for the Drive are STOPPED, SLOW, FASTER_THAN_SLOW, and CAR_IN_DECEL_ZONE (determined by commit point).
  • NEVER_ACCELERATED: A wasted opportunity for fast speed is detected if the drive never accelerates beyond slow speed between (non-leveling) stops.
  • PREMATURE_DECEL: A wasted opportunity for fast speed is detected if the drive decelerates from faster-than-slow speed before reaching the deceleration zone based on the commit point.


R-T10: For each stop at a floor, at least one door reversal shall have occured before the doors are commanded to nudge.

      The monitor enforces this by constructing two state machines, DoorState and ReversalState.
  • The Door state machine keeps track of if the doors are nudging or not.
  • The Reversal state machine keeps track of if a reversal has occured at that floor already.
  • If the doors are nudging, but the doors have not been reversed yet at that instance, then this violates our requirement, and a warning is thrown.


R-T11: Doors open and close are a continues process, there should not be any abrupt jump between the process:
  • When door is closed, the state shall be CLOSED
  • When door is open, the state shall be OPEN
  • When door is either opening or closing but not in a fully opened or closed state, the state shall be INTERMEDIATE.
  • When the door is at a fully open or closed state, and the sensor indicates an abrupt state jump to its opposite fully open or closed state, the state shall be UNDETERMINED.


R-T12: For the doors that are supposed to open, they should open in parallel
  • If only a single hallway's doors are to open or close, both side shall move in parallel such that there would not be a single moment that one door is closed while the other is open.
  • If both hallways' doors are to open or close, all doors shall move in parallel such that there would not be a single moment that one door is closed while the other is open.

Verification Instructions

Run the monitor with:
java simulator.framework.Elevator -monitor RuntimeRequirementsMonitor -pf #.pass
(where # is the name of the acceptance test file)
